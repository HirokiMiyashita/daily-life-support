# 減量プラン管理アプリ 要件定義 & データ仕様（半年-20kg / 朝ジムなし版）
（React Native + Expo / Supabase / 個人利用・TestFlight配布）

---

## 1. 目的 / ゴール
- 「半年で-20kg」を達成するための **毎日のやること（ToDo）・食事メニュー・ジムメニュー** を
  - **当日すぐ見れる**
  - **迷わず実行できる**
  - **記録して改善できる**
  状態に落とし込む。
- 追加ゴール：
  - **1週間分に必要な食材をサイドメニューで即確認**できる（買い物が迷わない）

---

## 2. ユーザー前提（今回の設定）
- 年齢: 26歳
- 仕事: 在宅 / 9:00〜18:00
- 身長: 158cm
- 体重: 76kg（開始時）
- 体型: 筋肉質 / 固太り
- ジム: 契約済み、利用可
- 方針: 朝ジムなし、夜ジム中心

---

## 3. 非公開アプリとしての利用形態（前提）
- 世に出す想定なし（ストア公開しない）
- 最終到達：
  - Expo（EAS）でビルド
  - TestFlight 経由で **本人のiPhoneに配布して動作確認・利用**
- 利用者は基本1人（本人）を想定
- 認証は最小（Supabase Authで匿名認証 or Email認証）
  - 個人利用でもSupabaseのRow Level Security (RLS)でセキュリティ確保

---

## 4. 技術要件（スタック確定）
### 4.1 フロント
- React Native（Expo）
- ナビゲーション：React Navigation（Drawerでサイドメニュー）
- 状態管理：必要最小（React Query or Zustand）
- オフライン/キャッシュ（任意）：AsyncStorage（のちに追加）

### 4.2 バックエンド / DB
- DB：Supabase（PostgreSQL）
- クライアント：Supabase JS Client（@supabase/supabase-js）
- API：
  - Supabaseの自動生成REST APIを直接利用
  - Row Level Security (RLS)でセキュリティポリシーを設定
  - 必要に応じてEdge Functions（Deno）でカスタムロジック実装
- 利点：
  - 認証機能が組み込まれている（Supabase Auth）
  - Realtime機能でリアルタイム同期が可能
  - ストレージ機能で写真保存も可能
  - スキーマ管理はSupabase Dashboard or マイグレーションSQL

### 4.3 認証（個人利用）
- Supabase Authを利用：
  - 方式A（推奨）：匿名認証（Anonymous Auth）
    - 初回起動時に自動で匿名ユーザーを作成
    - ログイン不要でシームレス
  - 方式B：Email認証（将来的に複数デバイス対応する場合）
    - メールアドレス + パスワード or Magic Link
- RLS（Row Level Security）設定：
  - 各テーブルにRLSを有効化
  - ポリシー：`auth.uid() = user_id` で自分のデータのみアクセス可能

## 5. 成果物（アプリが提供する価値）
### 5.1 Today画面（最重要）
- 今日の「やることリスト（チェック式）」
- 今日の「食事メニュー（朝/昼/間食/トレ後/夜）」
- 今日の「ジムメニュー（曜日別）＋有酸素設定」
- 今日の「目標（カロリー/タンパク/歩数/睡眠）」の表示

### 5.2 サイドメニュー（追加要件）
- **今週（または選択週）に必要な食材リスト（買い物リスト）** を表示
- 食材は「テンプレ食事」から自動集計
- 表示はカテゴリ別（例：タンパク / 野菜 / 糖質 / その他）
- 任意で以下ができると便利
  - [ ] 購入済みチェック
  - [ ] 数量の手動調整
  - [ ] 今週のメニュー変更（代替食材選択）に応じて再計算

### 5.3 記録
- 体重（毎朝）
- ウエスト（週1）
- 食事達成（メニュー通り/一部変更/未達）
- トレーニング実績（重量/回数/セット or 完了のみ）
- 歩数（手入力 or 将来連携）

### 5.4 停滞対応（提案）
- 「2週間平均が落ちない」などの条件で提案
  - 有酸素 +15分
  - 脂質 -10g
  - 炭水化物 -20g

## 6. 画面 / 機能要件

### 6.1 ホーム（Today）
- 今日の日付・曜日
- 今日のタイプ:
  - `TRAINING_DAY`（筋トレ＋有酸素）
  - `CARDIO_DAY`（有酸素Day）
  - `REST_DAY`（休養 or 任意散歩）
- セクション
  1) 今日のやること（チェックリスト）
  2) 今日の食事メニュー
  3) 今日のジムメニュー（該当曜日のワークアウト）
  4) 今日の目標サマリ（Kcal / P / 歩数 / 睡眠）

### 6.2 週間プラン（Week）
- 月〜日の一覧
- 各日のタイプ・メニュー概要
- タップで日詳細（Today相当）

### 6.3 食事（Meals）
- テンプレメニュー一覧（朝/昼/夜/間食/トレ後）
- 食材置き換え（鮭↔サバ、鶏むね↔赤身肉等）
- 「今週の買い物リスト」生成元となる

### 6.4 トレーニング（Workouts）
- 曜日別メニュー閲覧
- 種目ログ（重量/回数/セット）入力（任意）
- 進捗ルール（RIR2、12回余裕→次回+2.5kg）をメモ表示

### 6.5 記録（Logs）
- 体重ログ（毎日）
- ウエスト（週1）
- 写真（週1、任意）
- 7日平均（将来）

### 6.6 サイドメニュー（Drawer）
- メニュー例
  - Today
  - Week
  - Meals
  - Workouts
  - Logs
  - **Shopping List（今週の材料）** ←追加
  - Settings

### 6.7 Shopping List（今週の材料）
- 表示単位：
  - 今週（Mon-Sun）または「今日を含む7日」
- 集計ロジック：
  1) 対象週のDayPlanを取得
  2) その日のMealPlan（テンプレ＋override）を展開
  3) MealItemのingredient（食材）を集約
  4) 同一食材は単位換算して合算（できる範囲で）
- 表示形式（例）
  - タンパク：
    - 鶏むね：2.0kg
    - 鮭：5切れ
    - 卵：20個
  - 野菜：
    - 冷凍ブロッコリー：1袋
    - キャベツ：1玉
  - 糖質：
    - 米：適量（週◯g）
    - バナナ：7本
  - その他：
    - 豆腐：7丁
    - 納豆：7パック
- チェック機能（任意）
  - 食材ごとに `purchased=true/false`

- 20:00 トレ後摂取（プロテイン＋糖質少量）
- 20:30 夕食（糖質控えめ）
- 21:30 風呂＋ストレッチ
- 23:30 就寝

## 8. 食事テンプレ（メニュー固定）
### 8.1 朝食（200〜250kcal）
- プロテイン 1杯（20〜25g）
- ギリシャヨーグルト100g or ゆで卵1〜2個

### 8.2 昼食（500〜550kcal）
- 鶏むね/皮なしもも 200g（代替: 魚/赤身肉）
- 白米 100g（炊飯後）
- 野菜 両手2杯
- 味噌汁（油少なめ）

### 8.3 間食（100〜150kcal）
- プロテイン 1杯 or 低脂質ヨーグルト

### 8.4 トレ後（150〜200kcal）
- プロテイン 1杯
- バナナ半分 or 白米 50g

### 8.5 夕食（400〜450kcal）
- 魚（鮭/サバ）or 赤身肉 150〜200g
- 豆腐1丁 or 納豆1パック
- 野菜 両手2杯（鍋/蒸し）
- 米: 基本なし（必要なら 50gまで）

### 8.6 目標（トレ日）
- 1450〜1550 kcal / P 130g固定 / F 40g上限 / Cは昼＋トレ後中心

### 8.7 目標（休み日）
- 1300 kcal / 昼の米50g or なし / 散歩45〜60分

## 12. Shopping List（週次材料リスト）仕様（追加）
### 12.1 目的
- 「今週のメニュー」を元に、**買うべき食材と数量**を一覧化して迷いをなくす。

### 12.2 集計対象
- 週の定義：
  - A案：月曜始まり（Mon-Sun）
  - B案：今日を含む7日
- MVPはA案推奨（曜日トレと整合する）

### 12.3 集計の考え方（重要）
- `MealTemplate.items` は「表示用」
- 買い物リストは「食材単位」で集計したいので、別途 **Ingredientマスタ** を持つ。
- 例：
  - 「鶏むね200g」 → Ingredient: chicken_breast, qty: 200, unit: g
  - 「卵2個」 → egg, qty: 2, unit: piece
  - 「冷凍ブロッコリー 1袋」 → frozen_broccoli, qty: 1, unit: bag

### 12.4 単位（最小セット）
- `g` / `ml` / `piece` / `pack` / `bag` / `bottle` / `serving`
- MVPでは「単位換算」は無理しない
  - 例：「1袋」「500g」は別のままでもOK（必要なら後で統一）

### 12.5 置き換え（代替食材）
- MealItemに alternatives 表示はあるが、買い物リストで確定するには
  - 「今週は鮭→サバにする」など、週次の選択が必要
- MVP：
  - 代替選択は未実装でもOK（固定メニューで集計）
- P1：
  - 週ごとの ingredient override を持つ

### 12.6 購入チェック
- `ShoppingListItem.purchased` をbooleanで保持
- 表示ではチェックボックスにする

## 13. Supabase（DBスキーマ）に落とす時の考慮（概要）
- スキーマ管理：
  - Supabase DashboardのSQL Editorで直接実行
  - またはマイグレーションファイル（SQL）を管理
  - Prismaを使う場合も可（SupabaseはPostgreSQLなので互換性あり）
- 接続方式：
  - App → Supabase Client（HTTPS）→ Supabase REST API → PostgreSQL
  - Supabase URLとAnon Keyは環境変数で管理（EAS Secrets）
  - Service Role Keyは絶対にクライアントに含めない（サーバーサイドのみ）
- セキュリティ：
  - RLS（Row Level Security）で各テーブルのアクセス制御
  - Anon Keyは公開されてもRLSで保護されるため安全
  - "個人利用"でもRLSを設定することでセキュリティを確保
